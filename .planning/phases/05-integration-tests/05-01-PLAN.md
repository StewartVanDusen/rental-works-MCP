<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Test process -> Live API | Tests send authenticated requests to production RW instance |
| Environment -> Test file | Credentials read from env vars at process start |
| Test output -> stdout | Test results may contain API response data |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Information Disclosure | live-api.test.ts | mitigate | Never log raw JWT tokens or credentials in test output. Do not use `console.log(jwt)`. Assert on properties, not full objects. |
| T-05-02 | Tampering | live-api.test.ts | mitigate | All tests are strictly read-only: only `browse()`, `getById()`, `getSession()`, `authenticate()` calls allowed. No `create`, `update`, `remove`, `post` (except browse which uses POST internally), `put`, `delete` methods in test code. |
| T-05-03 | Information Disclosure | test output / CI logs | mitigate | Tests assert on field presence (`toHaveProperty`) not field values. No `expect(record.Email).toBe(...)` that would leak PII in test failure output. |
| T-05-04 | Information Disclosure | source control | mitigate | Credentials come from env vars, not hardcoded. No `.env` file committed. `describe.skipIf` prevents accidental runs without credentials. |
</threat_model>
