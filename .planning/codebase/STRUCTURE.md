## Directory Purposes

**src/**
- Purpose: All TypeScript source code
- Contains: MCP server, tool definitions, API client, types, utilities, tests
- Key files: `index.ts`, `tools/*.ts`

**src/tools/**
- Purpose: MCP tool definitions grouped by business domain
- Contains: 11 module files (one per RentalWorks domain), each exporting a registerXxxTools() function
- Key files: `inventory.ts`, `orders.ts`, `customers.ts`, `contracts.ts`, `billing.ts`, `vendors.ts`, `reports.ts`, `settings.ts`, `admin.ts`, `storefront.ts`, `utilities.ts`
- Pattern: Each file exports a single function `registerXxxTools(server: McpServer)` that calls `server.tool()` for each tool in that domain

**src/types/**
- Purpose: TypeScript interfaces for RentalWorks API types
- Contains: Single file `api.ts` with all API request/response types
- Key files: `api.ts`

**src/utils/**
- Purpose: Shared infrastructure and helpers
- Contains: API client, Zod schemas, formatting/validation functions
- Key files: `api-client.ts`, `tool-helpers.ts`

**src/__tests__/**
- Purpose: Test files co-located with source
- Contains: Unit and integration tests for tool registration, API paths, validators
- Key files: `tool-registration.test.ts`, `api-paths.test.ts`

**dist/**
- Purpose: Compiled JavaScript output (do not edit)
- Generated by: `npm run build` (runs tsc)
- Consumed by: `npm start` (runs node dist/index.js)

## Key File Locations

**Entry Points:**
- `src/index.ts`: Main server bootstrap, registers all tool domains, starts MCP server
- `dist/index.js`: Compiled entry point, executed via `npm start`

**Configuration:**
- `package.json`: Dependencies (only @modelcontextprotocol/sdk), scripts, metadata
- `tsconfig.json`: TypeScript compiler config (ES2022 target, Node16 module resolution, strict mode)
- No .env file in repo (credentials via RENTALWORKS_* env vars at runtime)

**Core Logic:**
- `src/utils/api-client.ts`: RentalWorksClient class, JWT authentication, HTTP methods
- `src/tools/*.ts`: 114 MCP tool definitions across 11 files

**Testing:**
- `src/__tests__/*.ts`: 5 test files covering registration, API paths, helpers
- Run with: `npm test` (vitest)

## Naming Conventions

**Files:**
- Tool modules: `camelCase.ts` (e.g., `inventory.ts`, `orders.ts`)
- Test files: `descriptor.test.ts` (e.g., `api-paths.test.ts`)
- Single purpose: One business domain per tool file, one utility per utils file
- Export naming: `registerXxxTools()` functions match domain name

**Functions:**
- Tool registration: `registerXxxTools()` (e.g., registerInventoryTools, registerOrderTools)
- Handlers: Tool names use `snake_case` (e.g., "browse_rental_inventory", "get_order")
- Helpers: `camelCase` with clear action verb (buildBrowseRequest, formatBrowseResult, withErrorHandling)

**Variables:**
- Private API client: `client` (singleton via getClient())
- Arguments object: `args` (Zod-validated input)
- API response data: `data`
- Errors: `error` or descriptive names in catch blocks

**Types:**
- Interfaces: `PascalCase` (JwtRequest, JwtResponse, BrowseRequest, BrowseResponse)
- Enums: `PascalCase` (e.g., search operators as enum string literal)
- Type aliases: `PascalCase` with semantic meaning (ToolResult)

## Where to Add New Code

**New MCP Tool (most common):**
- Primary code: Add handler to existing `src/tools/{domain}.ts` file
- Input schema: Define Zod schema inline or add to `src/utils/tool-helpers.ts` if reusable
- Implementation: Call `client.get()`, `client.post()`, etc. or use `client.browse()` for standard browse patterns
- Format: Use `formatEntity()` or `formatBrowseResult()` to format response
- Example: See `src/tools/inventory.ts` for 10+ tool examples

**New Tool in New Domain:**
- Create: `src/tools/{newdomain}.ts`
- Structure: Export `registerNewDomainTools(server: McpServer)` function
- Register: Import and call in `src/index.ts` (after line 42)
- Tests: Add import to `src/__tests__/tool-registration.test.ts` if registering new tools

**New Reusable Helper:**
- Location: `src/utils/tool-helpers.ts`
- Pattern: Export named function, document with JSDoc, include usage examples in comment
- Use case: Schemas (browseSchema), formatters (formatEntity, formatBrowseResult), wrappers (withErrorHandling)

**New API Type:**
- Location: `src/types/api.ts`
- Pattern: Add interface or type alias, document section with comment header
- Export: Named export, no default exports
- Use: Import in tool handlers and API client as needed

**New Test:**
- Location: `src/__tests__/{feature}.test.ts`
- Framework: vitest (describe, it, beforeAll, expect, Mock)
- Pattern: Set up server, register tools, connect client, run assertions
- Naming: Describe what's tested (api-paths.test.ts, tool-registration.test.ts)

## Special Directories

**dist/**
- Purpose: Compiled JavaScript output
- Generated: Yes (by `npm run build`)
- Committed: No (in .gitignore)
- Do not edit: All files are generated from src/

**node_modules/**
- Purpose: Installed dependencies
- Generated: Yes (by `npm install`)
- Committed: No (in .gitignore)
- Do not edit: Use `npm install` to update

- Generated: No (manually written)
- Committed: Yes

**src/node_modules/**
- Purpose: Internal node_modules (artifact from build/development)
- Generated: Possible (unclear why present in src/)
- Note: Not standard; typically node_modules lives at project root only
